# Assessment: SubAgentPlugin
**File:** src/plugins/SubAgentPlugin.ts
**Reviewed:** 2026-03-26
**Risk level:** Medium

## Bug Fixes
- [x] `onActivityReset` race condition on concurrent `executeTool` calls (lines 21-26, 93): The class-level comment acknowledges this: a second concurrent `ask()` call overwrites `this.onActivityReset`, leaving the first call without inactivity tracking. The comment says the orchestrator calls serially in practice, but this is an undocumented constraint that is not enforced. If the orchestrator is ever changed to run sub-agents in parallel (e.g. via `Promise.all`), the first call will silently lose its inactivity timeout. Consider using a per-call closure variable rather than the shared instance field. **Fixed: replaced shared `onActivityReset` field with a `Set<() => void>` (`onActivityResetHandlers`); each `executeTool` call registers/deregisters its own handler.**
- [x] `inactivityTimer` and `absoluteTimer` are not cleared if `this.agent.ask()` throws synchronously (lines 96-102): The `finally` block does clear them, so this is actually handled correctly. No issue.
- [x] `rejectTimeout` is initialised with a non-null assertion `!` (line 70): `rejectTimeout` is assigned inside the `Promise` constructor callback, which is synchronous — the assignment always runs before `rejectTimeout` is used. The `!` assertion is technically safe here but masks the pattern from static analysis tools. A cleaner approach would be to initialise it as `undefined` and guard its use, or restructure to avoid the non-null assertion. **Fixed: initialised with a no-op default `(_err) => {}` so the `!` assertion is no longer needed.**

## Refactoring / Code Quality
- [x] No `getSystemPromptFragment()` implementation: The plugin registers a sub-agent as a tool but provides the orchestrator with no instruction about when or how to use it. Other wrapper plugins (e.g. `WeatherPlugin`) include a `getSystemPromptFragment()` that guides LLM usage. At minimum, the `description` passed to the constructor could be surfaced there. **Fixed: added `getSystemPromptFragment()` returning `this.description`.**
- [ ] Timeout logic is complex enough to warrant unit tests: The inactivity-reset and absolute-timeout paths involve several interacting timers. The logic is correct, but without tests a future refactor is likely to introduce regressions. This is a code quality note rather than a defect. **Skipped: test authoring is outside the scope of this skill.**
- [ ] `onInit` stores a tool-call handler on the sub-agent (line 37): If `onInit` is called more than once, the handler is overwritten silently. This is consistent with how `HeadlessAgent.setToolCallHandler` presumably works, but worth noting if the plugin is ever re-registered. **Skipped: informational only; no actionable change within this module.**

## Security
- [x] `args.task` is passed verbatim to `this.agent.ask()` (line 67, 97): There is no length cap or content validation on the task string. An orchestrator LLM that has been prompt-injected could construct an arbitrarily large or malicious task string for the sub-agent. A maximum task length (e.g. 10 000 characters) would limit the blast radius without meaningfully restricting legitimate use. **Fixed: task strings exceeding 10 000 characters are silently truncated before being passed to `ask()`.**
- [ ] Sub-agent tool calls are forwarded to the parent agent's `tool_call` event (line 39): This means a sub-agent's tool invocations are visible to any parent-agent listener registered on `tool_call`. If sensitive tools are registered on the parent (e.g. a tool that exposes credentials), a malicious task string could potentially cause the sub-agent to invoke them via the forwarded event. This depends on how `HeadlessAgent` and `BaseAgent` route tool calls, and should be reviewed in the context of `HeadlessAgent.ts`. **Skipped: by-design forwarding for observability; requires HeadlessAgent review to assess true risk.**

## Performance
- [ ] A new `Promise` and two `setTimeout` calls are constructed on every `executeTool` invocation (lines 71-91): For sub-agents called frequently this is negligible, but the per-call allocation could be avoided by setting up timers once and resetting them rather than recreating them each call. **Skipped: negligible in practice; optimisation not warranted.**
- [ ] `this.agent.ask()` is not cancellable after a timeout fires (lines 97-102): When either timer fires, `timeoutPromise` rejects and `Promise.race` resolves, but `this.agent.ask()` continues running in the background. Depending on what the sub-agent is doing (e.g. a long LLM call), this can cause resource leaks. `HeadlessAgent` would need to expose an abort/cancel API to address this fully. **Skipped: requires cancellation API on HeadlessAgent; cannot be fixed within this plugin alone.**

## Consistency / Style Alignment
- [x] `name` field is set to `toolName` from the constructor (line 29): This means the plugin's name is the tool name, not a human-readable plugin identifier. Other plugins use a short descriptive name (e.g. `"Weather"`, `"Notes"`). If any code iterates plugins by `name` to display a list or route logging, sub-agent plugins will appear with their tool names rather than a consistent "SubAgent" label. **Fixed: `name` is now `"SubAgent"`; tool name is stored in a separate `toolName` field used by `getTools()` and `executeTool()`.**
- [x] No logging: Unlike most other plugins, `SubAgentPlugin` emits no log entries (no `logger` import). Adding at least a `logger.debug` at the start of `executeTool` and on timeout would make sub-agent activity visible in debug logs. **Fixed: added `logger` import and `logger.debug` at the start of `executeTool`.**

## Notes
- The most significant operational risk is the acknowledged `onActivityReset` race condition. The class-level comment is helpful but the constraint (serial orchestrator calls) should ideally be enforced at the call site or documented in `src/plugins/CLAUDE.md`.
- The backgrounded `this.agent.ask()` after timeout is a resource-leak concern that cannot be fixed within this plugin alone — it requires a cancellation API on `HeadlessAgent`.
- Cross-module: `onInit` calls `this.agent.setToolCallHandler(...)` on the sub-agent. Reviewers of `HeadlessAgent.ts` should verify that multiple calls to `setToolCallHandler` do not stack listeners.
