# Assessment: WebReaderPlugin
**File:** src/plugins/WebReaderPlugin.ts
**Reviewed:** 2026-03-28
**Risk level:** Medium

## Bug Fixes
- [ ] `executeTool` returns `undefined` for unknown tool names implicitly, but the function has no explicit `return undefined` — this is fine by convention (per Plugin CLAUDE.md) but is currently compliant only because there is a single tool. If a second tool is added in future and the condition is not extended, the silent undefined will mask routing errors. Not a current bug, but a latent one.
- [ ] Link deduplication uses `seen.has(href)` where `href` is the raw attribute value resolved by JSDOM (line 101–106). Because `seen.add(href)` stores the resolved absolute URL, two different anchor elements pointing to the same absolute URL will be correctly de-duplicated. However, `http:` links are included in the `links` array (line 104) even though `validateUrl` rejects `http:` URLs for fetching. This means the agent may be handed links it cannot actually use via `read_webpage`, creating a confusing mismatch between what is listed and what is fetchable.
- [ ] `res.text()` (line 87) is called unconditionally before checking `Content-Type`. For non-HTML responses (e.g., JSON APIs, binary files, PDFs), JSDOM will silently produce a malformed document and Readability will return `null`, yielding a generic `"Could not extract readable content"` error with no informative message. The original content type is lost.

## Refactoring / Code Quality
- [ ] Module-level mutable `let Readability` / `let JSDOM` variables (lines 5–6): lazy initialisation via bare module-level `let` is a simple pattern but is not thread-safe under concurrent calls. Two simultaneous `readWebpage` invocations can both see `null` and both `await import(...)`, resulting in two imports that then overwrite each other. The assignments are idempotent so this is unlikely to cause observable bugs, but the pattern should use a single cached promise (e.g., `let jsdomPromise: Promise<...> | null = null`) to be correct.
- [ ] `executeTool(name: string, args: any)` uses `any` for `args` (line 66). A typed interface or at minimum an inline type `{ url: string }` would catch call-site errors at compile time.
- [ ] The 8 000-character truncation threshold (line 122) is a magic number. It should be a named constant at the top of the file (e.g., `const MAX_CONTENT_LENGTH = 8_000`) to make the intent clear and allow easy tuning.
- [ ] `validateUrl` is a module-level free function, not a private method. Since it is only used by this class, it should either be a private static method or, if kept free-standing, placed directly adjacent to the class rather than at the very top of the file away from its only consumer.
- [ ] The `baseUrl` variable (line 95) is constructed from `new URL(url)` immediately after `validateUrl(url)` has already parsed the same URL and returned a `URL` object. The return value of `validateUrl` is discarded and the URL is parsed a second time. `validateUrl` should return the parsed `URL` and the caller should reuse it.

## Security
- [ ] SSRF via DNS rebinding: `validateUrl` checks the hostname string at request construction time. A DNS rebinding attack can return a public IP for the initial DNS lookup (passing the check) and resolve to a private IP for the actual TCP connection. This is a known limitation of host-based SSRF guards without redirect validation. While this is hard to fully fix in pure JS without OS-level controls, the risk should be documented and, where possible, mitigated by following redirects carefully or disabling redirect following (`redirect: "error"` in fetch) for sensitive deployments.
- [ ] `http:` links are included in the returned `links` array (line 104). If the agent naively passes these back to `read_webpage`, the call will fail with "Only HTTPS URLs are allowed." — but it also means the plugin actively surfaces non-HTTPS URLs to the LLM, which could be used in a prompt-injection scenario to downgrade link navigation to plaintext. Filtering to `https:` only in the links array is the safer default.
- [ ] No `Content-Type` validation is performed on the response (line 87). A malicious or misconfigured server could respond with `text/html` Content-Type but serve a gigantic binary payload, causing `res.text()` to buffer up to several hundred MB into memory before JSDOM processes it. A `Content-Length` or streaming size cap check should be added before calling `res.text()`.
- [ ] The `User-Agent` header (line 79) impersonates a real Chrome browser version. While common for scraping, this may violate some sites' terms of service and could also mislead server-side bot detection in unintended ways. Consider using a neutral agent string or making it configurable.

## Performance
- [ ] `res.text()` (line 87) buffers the entire HTML document in memory before any parsing begins. For large pages this can allocate many megabytes. A maximum byte limit check (e.g., inspecting `Content-Length` header before reading, or using a streaming reader with a byte counter) would bound memory use.
- [ ] JSDOM is a heavy DOM environment that executes no scripts here, but its instantiation cost for large documents is non-trivial. The `{ url }` option passed to JSDOM (line 92) triggers base-URL resolution but also enables resource loading hooks. Passing `{ url, resources: "usable" }` is not set, so external resources are not fetched — this is correct — but it is worth an explicit comment to signal the decision.
- [ ] `document.querySelectorAll("a[href]")` (line 98) iterates the entire document to collect links before Readability runs. Readability internally clones and traverses the document as well. For very large documents this means two full traversals. If link extraction performance becomes a concern, links could be extracted from the Readability clone or a DocumentFragment instead.

## Consistency / Style Alignment
- [ ] `getSystemPromptFragment()` is implemented but `getContext()` is not. For a plugin that may want to surface per-call state (e.g., last fetched URL), the lifecycle hook is available but unused. This is fine as-is, but worth noting for future extension.
- [ ] Error messages thrown by `validateUrl` (lines 13, 16, 31) are plain strings without a prefix or error code, which differs from some other plugins that use structured errors or logger calls. Consistent error formatting would help callers distinguish validation errors from network errors.
- [ ] Per the plugin CLAUDE.md conventions, `executeTool` should return `undefined` (not implicitly fall through) for unknown tool names. The current implementation falls through implicitly (line 66–70), which is correct in practice but should have an explicit `return undefined` to signal intent to future maintainers.
- [ ] `logger.debug("WebReader", ...)` (line 74) — the first argument appears to be a tag/namespace. Other plugins should be checked to confirm this is the standard call signature for `logger.debug`; if the signature is `(message: string)`, the tag argument is silently ignored.

## Notes
- The module depends on `jsdom` and `@mozilla/readability` as runtime dependencies. These are not validated at startup (deferred until first use). If either package is missing, the first `read_webpage` call will throw a dynamic import error. Consider adding an `onInit` hook that performs a test import and logs a warning if the packages are absent, rather than failing silently at call time.
- The 15-second fetch timeout (line 82) is generous for an interactive agent. If the agent is used in a latency-sensitive context, this could stall the conversation noticeably. Making the timeout configurable via a constructor option would improve composability.
- `WebSearchPlugin` and `WebReaderPlugin` are complementary and often registered together (see `createWebAgent`). A cross-plugin concern worth noting for reviewers of that factory: there is no shared rate-limiting or circuit-breaker between them.
