# Assessment: CortexMemoryPlugin
**File:** src/plugins/CortexMemoryPlugin.ts
**Reviewed:** 2026-03-28
**Risk level:** Medium

## Bug Fixes
- [ ] `executeTool` returns `undefined` for unknown tool names via fall-through, but also returns `undefined` when a known handler is reached but an internal error is thrown (the outer `catch` block swallows the error and returns `undefined` implicitly). The caller receives `undefined` rather than an error string, which can cause confusing silent failures. Each handler should return a user-facing error string on failure, or rethrow; the outer catch should return a string like `"Tool error: ..."` rather than `undefined`.
- [ ] `delete_memory` (line 350–354) does not verify the memory exists before attempting deletion. If the LLM supplies a non-existent ID, no error is surfaced — the operation silently succeeds. Add a `getMemoryById` check (as done in `edit_memory`) and return a descriptive message when not found.
- [ ] `onMessage` autonomous conflict resolution (line 462) concatenates `this.currentEvents.join(" ") + " " + content` to form the conflict query. If `currentEvents` is empty (e.g., first turn or after reset) the query is just the full assistant message, which may produce spurious high-similarity matches and incorrectly delete or supersede valid memories. Guard with a `currentEvents.length > 0` check or raise the similarity threshold when the event context is absent.
- [ ] `get_linked_memories` and `delete_memory` call `args.id.slice(0, 8)` (lines 357, 351) without validating that `args.id` is a non-empty string. If the LLM passes `null`, `undefined`, or a non-string, a runtime TypeError is thrown but caught silently by the outer `try/catch`, returning `undefined`. Add a type/presence guard at the top of each handler.

## Refactoring / Code Quality
- [ ] The `executeTool` method is 155 lines of `if (name === ...)` chains (lines 297–451). Extract each handler into a private method (e.g., `private handleSearchMemory(args)`, `private handleSaveMemory(args)`, etc.) and reduce `executeTool` to a dispatch table. This improves readability and makes individual handlers independently testable.
- [ ] The truncation constant `MAX_TEXT = 300` is duplicated in four separate handler blocks (lines 304, 377, 400, 439). Hoist it to a private readonly class field `private readonly MAX_MEMORY_TEXT_LENGTH = 300` and reference it from each handler.
- [ ] `constructor` accepts `llmProvider: any` and passes it directly to `CortexMemoryDatabase`. The `any` type removes all static-analysis safety. Define or import a minimal interface (e.g., `LLMProvider`) and use it here, consistent with how other plugins type their constructor arguments.
- [ ] `getSystemPromptFragment` (line 41) calls `this.db.getRecentMemories(20, "behavior")` synchronously. If this method ever becomes async (or if a database error occurs), the fragment silently returns without behavior entries. Add error handling around this call.
- [ ] `onInit` (line 16) is a no-op with a comment. Per plugin conventions (`CLAUDE.md`), if there is nothing to do, the hook can simply be omitted rather than implemented as an empty stub.
- [ ] The `currentEvents` field (line 9) is set in `getContext` (line 54) and read in `onMessage` (line 462), creating implicit temporal coupling between two lifecycle hooks. Document this coupling with a comment, or refactor to pass context explicitly.

## Security
- [ ] `args` values throughout `executeTool` are passed directly from the LLM without any input validation or sanitisation beyond type coercion via `String(...)`. For fields like `args.content`, `args.rule`, `args.goal`, and `args.steps`, there is no length cap enforced at the plugin level — an excessively large string could exhaust memory when embedding or storing. Enforce a maximum content length (e.g., 10 000 characters) and return an error string rather than attempting to process oversized inputs.
- [ ] `args.filter` in the `aggregate_memories` handler (line 413) is spread directly from user-supplied LLM output into a `MemoryFilter` without field validation. If the downstream `aggregateMemories` method does not validate its input, malformed filter fields could cause unexpected query behaviour. Validate field types before use.
- [ ] Date strings `args.after`, `args.before`, `args.start`, `args.end` (lines 369–370, 434–435) are passed to `new Date(...)` without checking for `Invalid Date`. An invalid date produces `NaN` as a timestamp, which may cause unintended query ranges or errors in the database layer. Validate with `isNaN(new Date(val).getTime())` before use.

## Performance
- [ ] In `save_memory` (line 320), after saving a memory, a second semantic search is run (`this.db.search(args.content, 3, 0.5)`) to find memories to link. This requires embedding the same content a second time. Reuse the embedding returned by `addMemory` (or returned alongside it) to avoid a duplicate embedding round-trip to the LLM provider.
- [ ] `getSystemPromptFragment` (line 41) calls `this.db.getRecentMemories(20, "behavior")` on every LLM call. If behavior memories change infrequently, this query is repeated unnecessarily. Consider caching the behavior list and invalidating it only when `save_behavior` or `delete_memory` is called for a behavior record.
- [ ] `onMessage` (line 464) calls `this.db.search(conflictQuery, 5, 0.85)` on every assistant message, requiring a full embedding + cosine-similarity scan of the memory store. On large memory stores this adds latency to every turn. Consider rate-limiting or sampling this check (e.g., only run when `savedThisTurn` is non-empty, since conflicts are most likely when new memories were just written).

## Consistency / Style Alignment
- [ ] `executeTool` is declared `async` (line 297) but several tool handlers (`query_memories`, `aggregate_memories`, `get_memory_timeline`) call only synchronous database methods and return plain values, not Promises. While harmless, the mixed sync/async handling within a single `async` method makes the control flow harder to reason about. Add `await` where missing or extract synchronous handlers into clearly synchronous private methods.
- [ ] The `onMessage` signature (line 454) includes `_source: string` with a leading underscore indicating it is unused — consistent with project convention. However, the `_agent` parameter in `onInit` (line 16) also uses the underscore prefix. Both are consistent; this is noted as good practice already applied.
- [ ] Log messages use the tag `"CortexMemory"` throughout, but the class `name` field is `"CortexMemory"` (line 7) while the class itself is `CortexMemoryPlugin`. All other plugins use their short class name as the log tag (e.g., `"ThoughtPlugin"`). Standardise to `this.name` to align with the project convention.
- [ ] `save_memory` returns `Memory saved (id: ${id.slice(0, 8)}).` while `save_behavior` returns `Behavior rule saved (id: ...)` and `save_procedure` returns `Procedure saved (id: ...)`. The phrasing is inconsistently capitalised and formatted across the three handlers. Align them to a single pattern, e.g., `Memory saved (type: behavior, id: xxxxxxxx).`

## Notes
- `CortexMemoryPlugin` is tightly coupled to `CortexMemoryDatabase` — it constructs the database directly and exposes `db` as a `public` field (line 8). The `ThoughtPlugin` also calls `db.addMemory` directly on this public field. Any reviewer of `ThoughtPlugin` should be aware of this cross-module write path and ensure it remains consistent with `savedThisTurn` tracking (it currently bypasses `savedThisTurn`, meaning thoughts saved by `ThoughtPlugin` will not be excluded from the conflict-resolution scan in `onMessage`).
- The autonomous memory deletion logic in `onMessage` (lines 468–479) is potentially destructive: it permanently deletes memories less than two hours old based solely on cosine similarity ≥ 0.85 with the current turn content. A high-similarity threshold reduces false positives, but reviewers of `CortexMemoryDatabase` should verify that the `search` method returns accurate scores and that the threshold cannot be trivially triggered by unrelated but lexically similar content.
