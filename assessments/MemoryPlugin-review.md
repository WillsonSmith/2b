# Assessment: MemoryPlugin
**File:** src/plugins/MemoryPlugin.ts
**Reviewed:** 2026-03-28
**Risk level:** Medium

## Bug Fixes
- [x] `getMessages` returns wrong result when `limit` is `0`: At line 57, the condition `if (historyLimit && historyLimit > 0)` treats `limit=0` as falsy, causing it to fall through to the full-history branch (`chatHistory = [...this.messages]`). A caller explicitly passing `0` to request no history would instead receive all messages. Fix: change the condition to `if (historyLimit !== undefined && historyLimit > 0)` (already accounting for the system-prompt subtraction above, but note `historyLimit` may never reach `0` due to the decrement at line 53, so the edge case is `limit=1` with a system prompt — that decrements `historyLimit` to `0` and then the full history branch fires). The correct guard is `if (historyLimit !== undefined && historyLimit > 0)`.
- [x] `summarizeOldContext` infinite-loop risk: At lines 83–88, the `while` loop scans forward for a `user` message. If all messages from `splitIndex` onward are `assistant` messages (e.g. because the last message was an assistant reply), `splitIndex` will advance past `this.messages.length` and the loop exits via the bounds check — but `recentMessages` (line 91) will then be empty, and the early-return at line 93 fires without ever trimming `this.messages`. The net effect is that `this.messages.length` stays above `MAX_MESSAGES` and every subsequent `onMessage` call will re-enter `summarizeOldContext` repeatedly. Fix: add a hard cap fallback that trims the array even when no user message boundary is found after the split point.
- [x] Duplicate JSDoc block: Lines 38–45 contain two `/** … */` JSDoc comments for `getMessages`. The first (lines 38–41) is a stale, incomplete version; the second (lines 42–46) is the accurate one. TypeScript tooling attaches the last doc-comment to the method, so this is not a runtime bug, but it pollutes IDE hover text and should be removed.

## Refactoring / Code Quality
- [x] Comment mismatch at line 68: The comment reads `// 4. Prepend the system prompt at the very end` but the operation is "prepend at the very beginning." Fix: change to `// 4. Prepend system prompt`.
- [x] `_source` parameter is unused and undocumented: `onMessage` accepts `_source` (line 22) but neither uses it nor stores it. The interface (`AgentPlugin`) provides this for plugins that need per-source routing. Either document why it is intentionally ignored or remove the leading underscore convention comment so readers understand the intentional discard.
- [x] Magic numbers for `MAX_MESSAGES` / `MIN_MESSAGES` are not constructor-injectable: Both constants (lines 14–15) are `private readonly` class fields with hardcoded defaults. Callers cannot tune summarization thresholds without subclassing. Consider accepting an optional config object `{ maxMessages?: number; minMessages?: number }` in the constructor to improve reusability.
- [ ] `summarizeOldContext` mutates `recentMessages[0]` before assigning to `this.messages`: Lines 106–109 mutate the local `recentMessages` array element in place before assigning the array. This is harmless now but fragile — if `recentMessages` were ever a live reference to `this.messages` rather than a slice, it would corrupt the original. The pattern is safe here but worth a clarifying comment or a more explicit clone.

## Security
- [ ] Summary content is injected verbatim into the next user message (line 108): The `summaryResponse` from the LLM is embedded directly into the content of the next `user`-role message without any sanitisation. A hostile assistant response could craft a summary that includes prompt-injection fragments (e.g. fabricated system instructions) which then re-enter the conversation context as apparently user-authored text. Fix: wrap the injected summary in a clearly delimited, read-only format and consider prepending it as a `system` message rather than appending to `user` content. At minimum add a note in the code that this content comes from the LLM.

## Performance
- [ ] `chatHistory.shift()` called in a loop (lines 64–66): `Array.shift()` is O(n) because it re-indexes every element. In the worst case (e.g. an assistant-heavy history after slicing), this loop runs multiple times with an O(n) operation each iteration, giving O(n²) behaviour. Fix: find the first user-message index with `findIndex` and call a single `slice` instead.
- [ ] `[...this.messages]` shallow copy at line 60: This is the correct approach to avoid aliasing, but the cost is paid on every call to `getMessages` when no limit is set. With `MAX_MESSAGES` capped at 15 this is negligible — no change needed, but worth a note if `MAX_MESSAGES` is ever increased significantly.

## Consistency / Style Alignment
- [ ] `onMessage` is declared `async` (line 19) but never `await`s anything in its own body — the only `await` is in the delegated call to `summarizeOldContext`. The async annotation is not wrong (it satisfies the `AgentPlugin` interface which types the return as `void | Promise<void>`), but it is inconsistent with how other simple hooks are written. Minor nit.
- [ ] Error logging at line 113 uses the positional `logger.error("Memory", …)` form: Other plugins use `logger.error` with a module tag matching the class name (e.g. `"MemoryPlugin"`). Use `"MemoryPlugin"` for consistency so log filters work uniformly.
- [ ] No `onInit` hook implemented: Per `plugins/CLAUDE.md`, I/O and setup should be deferred to `onInit`. The constructor only stores `llm`, which is fine, but this is worth noting for future changes — any new setup logic must go in `onInit`, not the constructor.

## Notes
- `MemoryPlugin` is the short-term/conversational memory layer and is distinct from `CortexMemoryPlugin` (long-term semantic memory). If both are registered on the same agent, `BaseAgent` must have a clear policy for which plugin's `getMessages` wins — that cross-module concern should be verified in `BaseAgent`.
- The summarization strategy (prepending a summary to the first retained user message) means summary text is attributed to the user role in the conversation. Downstream models that are sensitive to role-based trust will treat this LLM-generated summary as a user utterance. This design decision should be documented.
- `MIN_MESSAGES` (5) is the number of messages retained *after* summarization. If summarization itself fails, the fallback (line 117) also retains only those 5 messages, so context loss on LLM failure is bounded — this is good defensive behaviour worth preserving.
