# Assessment: stream-tts
**File:** src/utils/stream-tts.ts
**Reviewed:** 2026-03-28
**Risk level:** Medium

## Bug Fixes
- [x] Unhandled `ffplay` spawn failure (lines 23–40): `spawn()` does not throw if `ffplay` is not on PATH; it returns a subprocess whose `stdin` is `null` or immediately broken. The write at line 54 (`ffplay.stdin.write(...)`) will throw or silently drop audio without any user-facing error. Add a guard immediately after `spawn()` to check that `ffplay.stdin` is non-null and/or catch the subprocess exit code before proceeding.
- [x] `splitter.close()` not called on text-feed error (lines 69–77): The `for` loop that feeds tokens to `splitter.push()` has no try/catch. If an exception is thrown (e.g. the splitter is already closed or the underlying stream errors), `splitter.close()` at line 77 is never reached, leaving the background consumer goroutine blocked waiting for the stream to end and the `ffplay` stdin never closed.
- [x] `audio` truthy-check does not guard against empty typed arrays (line 50): `if (audio)` is true even when `audio` is a `Float32Array` of length 0, which results in writing an empty `Buffer` to `ffplay.stdin`. While usually harmless, it may produce audible glitches or cause ffplay to misparse the raw PCM stream.
- [x] `waveData.buffer` may include extra bytes (line 54): `Float32Array.buffer` is the underlying `ArrayBuffer`, which can be larger than the view if the typed array is a sub-view (non-zero `byteOffset` or `length < buffer.byteLength`). Using `Buffer.from(waveData.buffer)` instead of `Buffer.from(waveData.buffer, waveData.byteOffset, waveData.byteLength)` will write stale or zeroed bytes and corrupt the PCM stream.

## Refactoring / Code Quality
- [x] Module-level mutable singleton `ttsInstance` (line 5): The singleton is initialized lazily but never exposed for teardown or reset. If initialization fails mid-way (e.g. model download interrupted), `ttsInstance` remains `null` and each subsequent call retries initialization — but there is no retry limit or backoff. Extract initialization into a dedicated `initTTS()` function or add a separate `teardownTTS()` export so callers can manage lifecycle explicitly.
- [x] Artificial `setTimeout` delay simulates token streaming (lines 72–74): The 20 ms per-word delay is hardcoded and described only as "slightly faster" with no constant or configuration point. This makes the function unsuitable for real streaming (where tokens arrive asynchronously from an LLM). Replace the artificial loop with a parameter that accepts an `AsyncIterable<string>` of tokens, so the caller drives pacing rather than the utility.
- [x] `(audio as any)` double cast on line 52: The comment acknowledges an API difference between transformers.js versions, but the cast pattern `(audio as any).audio || (audio as any).data` is fragile. Define a local interface or type guard (`isAudioWithData`) to make the version fallback explicit and type-safe.
- [ ] `void` IIFE for background consumption (lines 43–66): Floating async IIFEs make error propagation and lifecycle management difficult. Consider returning a `Promise<void>` from the background task and attaching it to the returned subprocess object (e.g. as `ffplay.done`) so callers can await completion or attach their own error handlers. **SKIPPED** — attaching a `done` property to the Bun `Subprocess` type would require changing the return type from `Subprocess` to a wrapper object, which is an API-breaking change beyond safe scope.
- [x] Counter variable `i` declared but used only in debug log (line 44): The chunk index `i` is incremented unconditionally on `chunkText` presence, but since `i` is only used inside the `if (chunkText)` block, it will count text chunks, not audio chunks. Rename to `chunkCount` and document what it counts for clarity.

## Security
- [x] No sanitization of `text` input before feeding to splitter (lines 69–71): While the text is not passed to a shell command directly, arbitrarily long inputs will block the event loop for the duration of the token-feeding loop (20 ms × N words). Add a maximum input length guard or move the feeding loop to a worker/microtask queue.

## Performance
- [x] Synchronous per-token `await new Promise(r => setTimeout(r, 20))` (line 73): The 20 ms artificial delay is applied to every word serially, meaning a 100-word string takes ~2 seconds before `splitter.close()` is called and the final audio chunk can be generated. This is wasteful when the text is already fully available. At minimum, make the delay configurable (defaulting to 0 when text is pre-buffered).
- [x] `text.split(" ")` (line 69): Splitting on a single space does not handle multiple consecutive spaces or other Unicode whitespace, producing empty tokens that are pushed as `" "` strings, causing the splitter to receive spurious whitespace-only chunks. Use a regex split (`/\s+/`) and filter empty strings.

## Consistency / Style Alignment
- [x] `-ch_layout 1` ffplay argument (line 32): `1` is not a valid `ch_layout` string in FFmpeg — it should be `"mono"`. The current value may work on some FFmpeg builds that accept channel count integers but is not portable and deviates from FFmpeg's documented channel layout syntax.
- [ ] `logger.ts` import uses `.ts` extension (line 3): All other source imports in the project omit the `.ts` extension. Align with project convention by removing it: `import { logger } from "../logger"`. **SKIPPED** — `tsconfig.json` has `allowImportingTsExtensions: true` and the existing project style is mixed; this is a non-error and out of scope.
- [x] Inconsistent stdin guard: the background IIFE checks `if (ffplay.stdin)` before `ffplay.stdin.end()` (line 61), but the write at line 54 checks `if (waveData && ffplay.stdin)`. The outer `if (audio)` block on line 50 does not guard on `ffplay.stdin`, creating an inconsistency in the guard chain. Consolidate to a single early check at the top of the IIFE.

## Notes
- This module has a hard runtime dependency on `ffplay` being present on PATH. No check or helpful error message is produced if it is absent. Callers (e.g. AudioSystem) should document this requirement or the module itself should verify availability at initialization time.
- The `KokoroTTS` and `TextSplitterStream` types from `kokoro-js` are not typed in this file — all type information is inferred. If the `kokoro-js` package updates its API, type errors will only surface at runtime. Consider pinning the package version or adding explicit type assertions where the `any` casts occur.
- The function signature `runStreamingTTS(text: string): Promise<Subprocess>` returns before audio playback completes. Callers that need to await full playback have no way to do so without polling the subprocess exit code. This is a cross-module concern for any caller that needs ordered audio playback (e.g. queuing multiple TTS calls).
