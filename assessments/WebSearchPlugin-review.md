# Assessment: WebSearchPlugin
**File:** src/plugins/WebSearchPlugin.ts
**Reviewed:** 2026-03-28
**Risk level:** Medium

## Bug Fixes
- [x] `executeTool` returns `undefined` for unknown tool names instead of following convention: Line 31-35, `executeTool` has no explicit `return` for the case where `name !== "web_search"`. The plugin CLAUDE.md convention is "Return `undefined` (not throw) from `executeTool` for unknown tool names so other plugins can handle them." The implicit `undefined` return is correct in behaviour, but an explicit `return undefined;` in the else branch would make the intent clear and guard against future accidental implicit returns. Not a runtime bug, but a latent clarity issue. **Fixed: added explicit `return undefined;` for the non-matching branch.**
- [x] No error thrown if `args.query` is missing or not a string: Line 33, `this.search(args.query)` is called without validating that `args.query` is a non-empty string. If the LLM passes `{}` or `{ query: "" }`, `search` will fetch `https://api.duckduckgo.com/?q=&format=json&...`, silently returning an empty-results object. An empty query should be detected early and returned as a structured error rather than a wasted network call. **Fixed: `search` now accepts `unknown`, checks `typeof rawQuery !== "string" || rawQuery.trim() === ""`, and returns a structured error message without making a network call.**

## Refactoring / Code Quality
- [x] `name` field is `"WebSearch"` rather than `"WebSearchPlugin"`: Line 7, `name = "WebSearch"`. All other plugins in the codebase use `"<ClassName>"` format (e.g. `WikipediaPlugin`, `SubAgentPlugin`). Inconsistent naming makes plugin identification in logs and plugin-list iteration harder. Should be `"WebSearchPlugin"` to match conventions. **Fixed: `name = "WebSearchPlugin"`.**
- [x] `executeTool` signature uses `args: any` instead of the interface-specified `args: Record<string, unknown>`: Line 31. The `AgentPlugin` interface (Plugin.ts line 22) declares `executeTool(name: string, args: Record<string, unknown>)`. Using `any` disables TypeScript's type checking for args across the whole method body. **Fixed: signature is now `(name: string, args: Record<string, unknown>): Promise<unknown>`.**
- [x] `search` return type is untyped (`any`): Lines 37 and 54. The `private async search` method and its internal `data` variable are typed `any`. Defining an inline result shape or at minimum a return type annotation would improve safety and readability. **Fixed: added `SearchResult` interface and typed `data` as `Record<string, unknown>`.**
- [x] Dead `results` object pattern adds unnecessary indirection: Lines 55-89, an intermediate `results: Record<string, any>` accumulator is built up with conditional property assignments, then returned. This is harder to follow than constructing and returning the object directly. The pattern also makes it easy to accidentally skip a field. A cleaner approach would be to compose the result object in one expression. **Fixed: result is composed with spread + conditional inclusions in one `result` object literal.**

## Security
- [x] No length cap on the `query` parameter: Line 33, user-supplied or LLM-supplied queries of arbitrary length are passed directly to the DuckDuckGo URL. While URL length limits will cause a fetch failure rather than data corruption, an excessively long query (e.g. from a prompt-injection attack) wastes bandwidth and may reveal sensitive context to the external API. A maximum query length (e.g. 500 characters) should be enforced before the network call. **Fixed: query is truncated to `MAX_QUERY_LENGTH = 500` before the URL is constructed.**
- [x] Response data from DuckDuckGo is trusted and returned verbatim: Lines 57-80, fields such as `data.Answer`, `data.AbstractText`, and topic `.Text` are returned to the agent without sanitisation. If DuckDuckGo ever returns unexpected types (e.g. an object where a string is expected), the agent will propagate malformed data to the LLM. Coercing each field to string before inclusion (e.g. `String(data.Answer)`) would prevent this. **Fixed: all fields are guarded with `typeof` checks and coerced with `String()` before inclusion.**

## Performance
- [ ] `AbortSignal.timeout(10_000)` creates a new signal on every call: Line 49. This is the correct and idiomatic approach in modern JS/Bun — no issue with correctness — but worth noting that this allocates a new `AbortController` internally per request. For high-frequency usage this is negligible; the current 10 s timeout is appropriate.
- [ ] No caching of repeated identical queries: The plugin issues a fresh HTTP request for every `web_search` invocation, including exact duplicate queries within the same session. A simple in-memory `Map<string, object>` cache with a TTL (e.g. 60 s) would eliminate redundant external calls in multi-turn conversations where the same query is issued more than once. Not critical, but worthwhile for cost and latency.

## Consistency / Style Alignment
- [ ] Logging namespace inconsistency: Line 38, `logger.debug("WebSearch", ...)` uses the short name `"WebSearch"`. Other plugins in the codebase log with the full plugin identifier prefixed in brackets, e.g. `logger.debug("[WikipediaPlugin] searching: ...")` (WikipediaPlugin.ts line 74). Should use `"WebSearchPlugin"` as the namespace to align with convention and match the corrected `name` field.
- [ ] Missing `statusText` in HTTP error message: Line 52, `throw new Error(\`DuckDuckGo returned ${res.status}\`)` does not include `res.statusText`. WikipediaPlugin and other plugins in the codebase include both status code and status text (e.g. `\`Wikipedia search failed: ${res.status} ${res.statusText}\``). The omission makes error messages less informative.
- [ ] `User-Agent` header differs from other plugins: Line 48, the header is `"2b-agent/1.0"`. WikipediaPlugin uses `"2b-agent/1.0 (https://github.com/WillsonSmith/2b)"`. The WebSearchPlugin header should include the repository URL to be consistent.

## Notes
- The DuckDuckGo Instant Answers API is a best-effort, undocumented endpoint. It frequently returns empty results for anything that is not a well-known entity or definition. The `getSystemPromptFragment` already attempts to communicate this limitation, but the system prompt could be more explicit that the tool is unsuitable for open-ended web search and should be complemented by `WebReaderPlugin` when a source URL is needed.
- No cross-module dependencies beyond `AgentPlugin`, `ToolDefinition`, and `logger` — this plugin is self-contained and safe to refactor independently.
