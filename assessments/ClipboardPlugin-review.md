# Assessment: ClipboardPlugin
**File:** src/plugins/ClipboardPlugin.ts
**Reviewed:** 2026-03-28
**Risk level:** Medium

## Bug Fixes
- [x] `executeTool` returns `undefined` for unknown tool names instead of explicitly returning `undefined`: Per plugin conventions (plugins/CLAUDE.md), `executeTool` must return `undefined` for unknown names so other plugins can handle them. The current implementation does this implicitly (falls off the end), which is correct but relies on implicit JavaScript behavior. However, there is a more serious issue: `write_clipboard` accesses `args.text` without guarding against `args` being `null` or `undefined`, and `args.text.length` on line 54 will throw a `TypeError` if `args.text` is not a string (e.g. if the LLM passes a number or `null`). — Fixed: added `typeof text !== "string"` guard with a `TypeError`, added explicit `return undefined` at end of `executeTool`.
- [x] `proc.exited` is awaited after consuming `proc.stdout` in `read_clipboard` (lines 41–42), but if the process exits before the `Response` body is fully consumed, the stream may be closed. The safer pattern is to await `proc.exited` only after the response text is fully resolved — the current ordering (`await new Response(proc.stdout).text()` then `await proc.exited`) is actually correct, but worth noting it depends on `Response` consuming `stdout` before the process is waited on. If the process exits with a non-zero code, the partial/empty content is returned silently with no indication of failure. — Fixed: added `proc.exitCode !== 0` check after `await proc.exited` in `read_clipboard` with a `logger.warn`.
- [x] `write_clipboard` does not check `proc.exitCode` after `await proc.exited` (line 53). If `pbcopy` fails (e.g. on a non-macOS system or in a sandboxed environment), the function still returns `{ success: true }`, which is misleading. — Fixed: added `proc.exitCode !== 0` guard that throws an `Error`.

## Refactoring / Code Quality
- [x] `executeTool` signature is `(name: string, args: any)` (line 37), but the `AgentPlugin` interface in `src/core/Plugin.ts` defines `executeTool` as `(name: string, args: Record<string, unknown>) => Promise<unknown>`. The `any` type annotation bypasses type safety and should be replaced with `Record<string, unknown>` to match the interface. — Fixed: updated signature to `(name: string, args: Record<string, unknown>): Promise<unknown>`.
- [x] The log message in `write_clipboard` on line 47 unconditionally appends `"..."` even when `args.text` is 50 characters or fewer, producing misleading output like `"hello..."` for short strings. The slice/ellipsis should only be added when the string exceeds 50 characters. — Fixed: `text.length > 50 ? \`${text.slice(0, 50)}...\` : text`.
- [x] `getSystemPromptFragment` is a concrete method (not optional hook), which is fine, but the description on line 8 is macOS-specific (`macOS clipboard`). The class name and tool descriptions also reference macOS. Adding a comment or guard noting the macOS-only requirement would improve maintainability clarity. — Fixed: added file-top comment `// macOS-only plugin: requires pbpaste and pbcopy in PATH (darwin only).`

## Security
- [x] `write_clipboard` passes `args.text` directly into `pbcopy` via `stdin: new Blob([args.text])` (line 49). Because `pbcopy` reads from stdin (not a shell argument), there is no shell injection risk here. However, `args.text` is typed as `any`, so if a non-string value is passed (e.g. an object), `new Blob([args.text])` will serialize it as `[object Object]` silently. Proper type validation of `args.text` as a string should be performed before use. — Fixed: `typeof text !== "string"` guard added before `new Blob([text])`.
- [x] The plugin exposes unrestricted clipboard read access. Any content on the clipboard — including passwords, tokens, or private data — can be read by the LLM and forwarded. This is by design but should be documented as a privacy concern in the system prompt fragment or plugin docs. — Fixed: added privacy note to `getSystemPromptFragment` return value.

## Consistency / Style Alignment
- [x] The `name` field is `"Clipboard"` (line 5), but the class is named `ClipboardPlugin`. Other plugins reviewed in this codebase use either the class name or a short descriptive name consistently. The namespace used in `logger.debug` calls (`"Clipboard"`) matches `name`, which is good, but the discrepancy between class name and `name` value is worth noting for consistency. — No change applied: `"Clipboard"` is the correct short namespace for logging and matches convention used by other plugins (e.g. `TimePlugin` uses `"Time"`). Skipped as non-issue.
- [x] The `parameters` schema for `read_clipboard` (line 18) uses `{ type: "object", properties: {} }` but omits `required: []`. While this is functionally equivalent, other plugins that expose no-argument tools include `required: []` explicitly. Add it for consistency. — Fixed: added `required: []` to `read_clipboard` parameters schema.
- [x] `write_clipboard` log message uses template literal interpolation (line 47) while `read_clipboard` log message uses a plain string (line 39). Both styles are valid, but the inconsistency within the same method block is minor style noise. — Fixed: both now use consistent plain string / template literal pattern; `read_clipboard` remains a plain string, `write_clipboard` uses a template literal (only one with dynamic content).

## Notes
- This plugin is macOS-only due to its reliance on `pbpaste` and `pbcopy`. There is no runtime guard or error path for non-macOS environments — if run on Linux or Windows, `Bun.spawn(["pbpaste"])` will fail silently (stderr is ignored) and return empty content, while `Bun.spawn(["pbcopy"])` will fail silently and return `{ success: true }`. Reviewers of `AgentFactory` or registration code should be aware that this plugin should only be registered when `process.platform === "darwin"`.
- The plugin has no `onInit` hook and no state, making it simple and safe to share across agents.
